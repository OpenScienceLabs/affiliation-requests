# Affiliation Criteria

This page describes the evidence OSL reviewers look for when evaluating a
project affiliation request.

## Required criteria

A project must meet all baseline requirements before affiliation can be
approved.

1. **Mission fit**: the project has a clear open-source, open-science, research,
   education, public-interest, or open-technology purpose.
2. **OSI-approved license**: the project uses an OSI-approved open-source
   license.
3. **Public license file**: the repository includes a public `LICENSE` file or
   clear equivalent.
4. **Code of Conduct**: the project includes `CODE_OF_CONDUCT.md` or an
   equivalent public Code of Conduct with a reporting path.
5. **Reachable maintainers**: at least one maintainer is active or reachable
   through a documented public contact path.
6. **Public repository**: the project repository is public and accessible.
7. **Public collaboration path**: the project has a public issue tracker,
   discussion channel, project board, mailing list, or documented contact path.
8. **Contributor safety**: the project is not archived, closed-source,
   abandoned, misleading, hostile, or unsafe for contributors.
9. **Independence understood**: maintainers understand that affiliation does not
   transfer ownership, governance, intellectual property, roadmap control, or
   maintenance responsibility to OSL.
10. **Acknowledgement**: if accepted, maintainers will acknowledge OSL
    affiliation in `README.md` and public documentation where applicable.

## Recommended criteria

These items are not always required, but they help reviewers understand project
readiness and community health:

- a README that explains purpose, users, and current status;
- contribution instructions;
- basic user and developer documentation;
- roadmap, project ideas, starter issues, or labels for contributors;
- a `SECURITY.md` file or vulnerability reporting path where relevant;
- recent releases, commits, issue responses, or a clear stable-maintenance
  status; and
- a clear explanation of the collaboration the project seeks through OSL.

## Reasons a request may be declined

OSL may decline or defer a request when evidence is missing or when affiliation
would be misleading or unsafe. Common reasons include:

- no OSI-approved license;
- missing public license file;
- missing Code of Conduct or missing reporting path;
- no reachable maintainer;
- private, archived, abandoned, or unreachable repository;
- project purpose is not aligned with OSL;
- unsafe or hostile community practices;
- misleading use of the OSL name or branding;
- unclear ownership, license, or intellectual-property status;
- project is not ready for contributors;
- application lacks evidence and maintainers do not respond; or
- affiliation would imply endorsement, certification, or support that OSL cannot
  provide.

Declining a request does not have to be permanent. Projects may apply again after
addressing the review points.

## Reasons affiliation may be paused or removed

An affiliated project may be marked at risk, paused, or removed from public OSL
lists if it later fails to meet baseline expectations. Reasons include:

- the repository becomes private, archived, unreachable, or closed-source;
- the project removes or changes its license in a way that is incompatible with
  affiliation;
- the Code of Conduct or reporting path disappears;
- maintainers are unreachable after repeated review pings;
- project links or metadata are misleading or stale;
- project communication becomes unsafe or hostile;
- the project implies OSL ownership, certification, sponsorship, or control; or
- maintainers request removal.

Removal is not necessarily permanent. A project may request reactivation after
restoring baseline requirements and maintenance.

## Program-readiness criteria

Affiliation alone does not make a project ready for GSoC, internships, grants,
or similar limited programs. When a project seeks program participation,
reviewers may also look for:

- available mentors with enough time to review contributor work;
- scoped ideas suitable for the program timeline;
- clear public issues, milestones, or project boards;
- documented development setup and contribution process;
- communication channels for contributors;
- capacity to review pull requests and support contributors; and
- realistic expectations about limited slots and external selection.

OSL will do its best to participate in GSoC where possible, but no project idea,
mentor, applicant, contributor slot, funding, internship, grant, or sponsorship
is guaranteed.

## Acknowledgement requirement

Accepted projects should add an acknowledgement to their README and public docs
where applicable. Suggested wording is available in
[acknowledgement.md](acknowledgement.md).

Projects should remove or update the acknowledgement if affiliation is paused,
removed, or changed.
